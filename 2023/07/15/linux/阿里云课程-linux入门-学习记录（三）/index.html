1.linux服务器运维基本操作

一、配置ssh免密登录

原理：
    1.首先在客户端C生成一对密钥 ssh-genkey -t rsa
    2.将公钥拷贝给服务端S ssh-copy-id 服务端S的ip（首次登陆需要密码）
    3.当C向S发送连接请求时（用户名+ip），C会用公钥进行加密，S进行解密，然后随机生成字符串给C，C将解密后的字符串发给S，二者完成互相确认身份，S允许C免密登录

二、linux文件上传与下载：sftp和lrzsz

三、挂载mount命令

挂载是指将一个设备挂载到一个已存在的目录上，要访问设备中的文件，通过访问这个目录来实现
命令格式：
    mount [-t vfstype] [-o options] device dir
    -t vfstype：指定文件系统的类型
    -o options：描述设备的挂载方式  
        loop：当成硬盘分区挂载
        ro：采用只读方式挂载
        rw：采用读写方式挂载
    device：要挂载的设备
    dir：要挂在的目录

四、系统服务管理

service --status-all：查看系统所有的后台服务进程
service 服务名 status：查看指定的后台服务进程的状态
service 服务名 stop/start/restart：将指定的后台服务关闭/启动/重启

chkconfig 服务名 on/off：让后台服务开机自启/不要自启

五、网络管理

主机名配置：
    hostname：查看主机名
    hostname 新名字：修改主机名（重启后无效）
    vim /etc/sysconfig/network：重启后永久生效
ip地址配置：
    vim /etc/sysconfig/netwok-scripts/ifcfg-eth0
    service network restart：重启网络服务
域名映射：
    vim /etc/hosts：将主机名和ip地址映射写入
网络端口监听
    netstat -nlpt 获取进程名、进程号、用户ID

六、定时命令crontab

crontab：用于设置周期性被执行的命令，可以在固定时间间隔执行指定的系统指令或shell脚本
service crond start/stop/restart/status

crontab [-u user] file
crontab [-u user] [-e | -l | -r]
    -u user：用来设定某个用户的crontab服务
    file：命令文件，表示将file作为crontab的任务列表文件并载入crontab
    -e：编辑crontab文件内容
    -l：显示crontab文件内容
    -r：删除定时任务配置
配置说明：
    *   *   *   *   *   command
    分  时  日  月  周  命令
配置实例：
    */1 * * * * date >> /root/date.txt
    每分钟执行一次date命令
    30 21 * * * /usr/local/etc/rc.d/httpd restart
    每晚21:30重启apache

2.LVS负载均衡实战——Linux Virtual Server

一、集群和分布式

集群：
    分为三种类型：
        LB：
            负载均衡，多个主机组成，每个主机只承担一部分访问请求
        HA：
            高可用，避免单点故障
            A = 正常时间 / (正常时间+故障时间)，有0.99,0.999,0.9999,0.99999等标准
            SLA：
                服务等级协议，是服务提供商与用户间定一个一种双方认可的协定，一般就通过三个9、四个9等来表示
        HPC：
            高性能
集群与分布式：
    集群：
        同一个业务系统，部署在多台服务器上。集群中，每一台服务器实现的功能没有差别，数据和代码都是一样的
    分布式：
        一个业务被拆分成多个子业务，部署在多态服务器上。分布式中，每一服务器实现的功能是有差别的，数据和代码也是不一样的。

    分布式是以缩短单个任务的执行时间来提升效率的，而集群则是通过提高单位时间内执行的任务数来提升效率。
分布式系统：
    分布式存储：Ceph、GlusterFS、MogileFS
    分布式计算：hadoop、Spark

二、Linux Virtual Server简介

LVS介绍：
    LVS，Linux Virtual Server，负载调度器，内核集成，阿里的四层SLB（server load balance）是基于LVS+keepalived实现的

LVS集群中的术语：
    VS：Virtual Server，用于负载均衡调度
    RS：Real Server，集群中的实际工作服务器
    CIP：Client IP，客户端IP
    VIP：Virtual Server IP，VS面向外网的IP
    DIP：Director IP，VS面向内网的IP
    RIP：Real server IP，实际工作服务器的IP
    访问流程：
        CIP <--> VIP == DIP <-->RIP

LVS工作模式：
    lvs-nat：修改请求报文的目标IP，多目标IP的DNAT（动态NAT）
    lvs-dr：操纵封装新的MAC地址
    lvs-tun：在原请求IP报文之外新加一个IP首部
    lvs-fullnat：修改请求报文的源IP和目标IP

    lvs-nat模式：
        本质是多目标IP的DNAT，通过将请求报文中的目标地址和目标端口修改为某RS的RIP和PORT实现转发
        1）RIP和DIP建议放在同一个IP网段，使用私网地址，RS的网关要指向DIP
        2）请求报文和响应报文都必须经由VS转发，VS易成为系统瓶颈
        3）支持端口映射，可修改报文的目标PORT
        4）VS必须是Linux系统，RS可以是任意OS系统
    ![lvs-nat原理图](lvs-nat原理图.png)
        lvs在内核中工作的阶段是2和3：
            ![lvs内核工作阶段](lvs内核工作阶段.png)
    
    lvs-dr模式:
        Direct Routing，直接路由，LVS默认模式，应用最为广泛。通过为请求报文重新封装一个MAC首部进行转发，源MAC是DIP所在接口的MAC，目标MAC是被选中的RS的RIP所在接口的MAC。
        优点：
            DR模式，在回复包时，不经过VS，减少了VS负载
        缺点：
            数据报链路层以上都不修改，所以源/目标的IP/PORT都不改变，无法进行端口映射，即客户机要访问80端口，目标机只能让出80端口被访问
        计网知识遗忘：
            数据报经过路由器的时候，源IP和目标IP是不会发生变化的，只有MAC会变化；只有NAT转换才会改变源/目标IP
    ![lvs-dr模式原理图](./lvs-dr模式原理图.png)

        DR模式的特点：
            1）VS和各RS都配置有VIP
            2）确保前端路由器将目标IP为VIP的请求报文发往VS
                由于RS上既有物理网卡IP，又配置了VIP，所以会与VS的VIP产生地址冲突，有以下三种方式来避免地址冲突：
                    i)在前端网关做静态绑定VIP和VS的MAC地址
                        缺点是静态绑定，一旦换了路由器，就要重新绑定，不常用
                    ii)在RS上使用arptables工具——也不常用
                    
                    iii）在RS上修改内核参数，以限制arp通告及应答级别————常用
                    
                    修改以上几个文件，可以使VIP称为RS的私有IP，外界不知道，只有VS知道

            3）RS的RIP可以使用私网地址，也可以是公网地址；RIP与DIP在同一IP网络；RIP的网关不能指向DIP，以确保响应报文不会经由VS
            4）RS和VS要在同一个物理网络
            5）请求报文要经由VS，但响应报文不经由VS，而由RS直接发往Client
            6）不支持端口映射
            7）无需开启ip_forward，参考[!https://www.jianshu.com/p/134eeae69281]
            8）RS可使用大多数OS系统

            物理网卡eth0（实际网卡）、回环网卡lo（逻辑上的设备，地址稳定），参考：[!https://www.cnblogs.com/chenpython123/p/10649718.html]

    ![lvs-dr通信示例](lvs-dr通信示例.png)

    lvs-tun模式：
        相比于DR模式，TUN模式不要求DIP和RIP在同一网段
        转发方式：
            不修改请求报文的IP首部（源IP为CIP，目标IP为VIP），而在原IP报文之外再封装一个IP首部（源IP是DIP，目标IP是RIP），将报文发往挑选出的目标RS；RS直接响应给客户端【源IP是VIP（所以每个RS也都有VIP），目标IP是CIP]
        ![lvs-tun模式原理图](./lvs-tun模式原理图.png)

        TUN模式特点：
            1）RIP和DIP可以不处于同一物理网络，即集群节点可以跨互联网连接，DIP VIP RIP可以是公网地址
            2）RS的tun接口上需要配置VIP地址，一遍接收VS转发过来的数据包，以及作为响应报文的源IP
            3）VS转发给RS时需要借助隧道，因此RS的OS需要支持隧道功能
            4）不支持端口映射
            5）请求报文经过VS，响应由RS自己完成
        应用场景：
            TUN模式常用来负载调度缓存服务器组，这些缓存服务器一般放置在不同网络环境中，可以就近折返给客户端
            LAN环境一般用DR模式，WAN环境虽然可以用TUN模式，但是更常用nginx/DNS等实现，因此TUN模式实际应用很少

LVS调度算法：
    分为静态方法和动态方法

    静态方法：
        1）RR:roundrobin，轮询，较常用
        2）WRR：Weighted RR，加权轮询，较常用
        3）SH：Source Hashing，源IP地址hash，实现session sticky；将来自于同一个IP地址的请求始终发往第一次挑中的RS，从而实现会话绑定
        4）DH：Destination Hashing，目标地址hash；第一次轮询调度至RS，后续将发往同一个目标地址的请求始终发至第一次挑中的RS；典型适用场景是正向代理缓存场景中的负载均衡，如：Web缓存

    动态方法：
        主要根据每个RS当前的负载状态及调度算法，将overhead=value值较小的RS进行调度
        1）LC：least connections 适用于长连接应用
        
        2）WLC：Weighted LC，默认调度方法，较常用
        
        3）SED：Shortest Expection Delay，初始连接高权重优先，只检查活动连接，不考虑非活动
        
        4）NQ：第一轮均匀分配，后续SED
        5）LBLC：Locality-Based LC，动态的DH算法，
        6）LBLCR：LBLC with Replication，带复制功能的LBLC，解决LBLC负载不均衡问题，从负载重的复制到负载轻的RS，实现web cache等

LVS相关软件：
    使用ipvsadm软件包实现相关管理，包括基础配置，RS的增删改查等
